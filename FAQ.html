The Basics

What is baitsquatting?
An attacker registers the intuitive bare-name alias of a scoped npm CLI tool, then waits for AI coding agents to recommend it at scale. No hallucination required — the AI follows the tool's own documentation.

Why didn't the permission prompt protect me?
It did — for a developer. The prompt showed the exact command: npx lhci autorun. A developer would recognise that lhci is not @lhci/cli.

The gap is that Claude Code is actively marketed to non-developer users ("vibe coders") who have no way to evaluate a bare npm package name at the moment of execution. The control was designed for developer judgment — it was not designed for delegated trust.

Scope & Severity

How broad is the attack surface?
After testing 45 packages: specific, not broad. Most popular CLI tools own their bare npm namespaces. The gap only exists when all four conditions are true:

#ConditionExample
1Tool migrated to scoped packaging@lhci/cli
2CLI binary kept the bare namebinary: lhci
3Docs teach bare invocation as primary patternnpx lhci autorun
4Bare npm namespace unclaimed or held by third partylhci → AnupamAS01

Tools like eslint, prettier, vercel, turbo, firebase-tools own their bare npm names — no gap.

Exploitation would require a model to drop the scope prefix, which is a model output error rather than doc-following behaviour. The namespace gaps are real and all teams were notified, but they are listed here for completeness rather than as confirmed baitsquatting cases.

Responsibilities

Is this the AI's fault?
For the confirmed Flow B cases — no. That's precisely what makes baitsquatting interesting. Claude Code is not making an error. It recommends the scoped form (npx @sentry/cli, npx @biomejs/biome) for tools whose docs use the scoped form. It recommends the bare form only where the tool's own documentation teaches it. The adjacent Flow A cases are different — those would require the model to drop a scope prefix the docs teach correctly, which would be a model error. Slopsquatting subset. No such output has been observed; those cases are reported for completeness.

Responsibility sits at four levels:
PartyResponsibility
Tool maintainerPrimary — created and perpetuated the namespace gap by not claiming the bare alias
AI platformSecondary — permission prompt designed for developer judgment; deployed to non-developer and agentic users who cannot evaluate bare package names at the point of execution
npm registryTertiary — no proactive detection of binary name / namespace collisions between scoped packages and bare aliases
Developer / vibe coderShared — verify namespace ownership before approving AI-proposed bare npx commands; use scoped names in CI configs and agent instructions; do not delegate trust without tooling to back it
Didn't some maintainers call this spam?
Yes — and the criticism had two legitimate dimensions worth separating.

First, the spam flag itself: several issues were filed in rapid succession across multiple repositories. This was assisted by AI tooling (Cowork) used to coordinate simultaneous outreach — a pattern that looks automated to a maintainer receiving it, regardless of intent. That's a fair observation about the process.

Second, the urgency framing: Flow A library packages (MUI, Redux, tRPC, Testing Library) received public GitHub issues implying urgency equivalent to the live incident. Those teams correctly pushed back — they can't unilaterally claim namespaces, and their packages have no npx execution path. Public issues were the wrong channel for Flow A library packages. All were subsequently updated with a scope clarification and closed.

What's worth noting separately: several maintainers had already identified the same risk and taken mitigating action before this outreach reached them. Clerk claimed clerk-nextjs silently. Aikido Security had claimed depcruise as a defensive placeholder in July 2025. The Biome team had been actively seeking an npm transfer for biome for some time. npm itself placed a security hold on supabase-js in January 2026 — before this research began. This is independent corroboration that the risk is real and recognised — not something that needed a researcher to surface it.

What To Do

I'm a tool maintainer — should I be worried?
Check these four conditions. If all are true, you have a Flow B gap:
  1. Your package is scoped (e.g. @org/cli)
  2. Your CLI binary has a bare name (e.g. mycli)
  3. Your docs teach npx mycli as the primary invocation
  4. The bare mycli npm namespace is unclaimed or held by an unrelated party
If only some conditions apply → likely Flow A only (lower severity).
If none apply → you're probably fine.
What's the systemic fix?

Three independent parties need to act. The highest-leverage single change is at the registry layer — it scales to every current and future case without requiring individual tool maintainers to act first.

LayerFixWhoLeverage
npm registry ★Detect: scoped package binary name ≠ registered bare namespace → alert maintainer proactively. One engineering change closes the structural gap across the entire registry.npmHigh — scales to all current and future cases

I'm a developer — how do I protect myself?
  1. Always use the fully scoped package name in npx commands: npx @lhci/cli autorun not npx lhci autorun
  2. Check who owns the bare namespace before running any AI-proposed npx command: npm view <package> maintainers
  3. In CI pipelines, pin to scoped package names in your scripts — never rely on bare binary aliases
  4. Audit your project's dependency tree: generate an SBOM and review any bare-name packages against their scoped equivalents — if the bare name isn't owned by the same maintainer as the scoped package, treat it as a risk
I'm a vibe coder / non-developer — what should I do?

When an AI coding agent proposes a shell command containing npx, pause before approving. Ask the agent: "Is this the official scoped package, or a bare alias?" A well-calibrated agent will clarify. If it can't, ask it to use the fully scoped form instead.

The permission prompt shows the exact command — but it can't tell you whether the npm package behind it is owned by the tool's maintainer or a stranger. That distinction is now yours to ask about.

The /npm-trust-check slash command for Claude Code will run the check for you. For AI-scaffolded projects, use the /sbom-generate slash command to generate a full dependency inventory — it will flag any bare-name packages that don't match their scoped equivalent. Improvements welcome.

Research & Disclosure

Who else validated this?
Three independent methodologies converged on the same namespaces:
  • AI-agent probing — live incident + cross-tool testing across Claude, Copilot, ChatGPT, Replit Agent, Gemini
  • Systematic README enumeration — Anupam Singh scanned ~170 GitHub organisations and 7,000+ repositories, identifying the same bare-alias collision pattern independently. Five new confirmed collisions found.
  • Hallucinated-package cross-check — Prof. Murtuza Jadliwala and Joe Spracklen (USENIX Security 2025 Distinguished Paper) shared their hallucinated JS package dataset. 5 names appear in both their hallucinated list and the BSQT bare-alias inventory. The names models most frequently hallucinate are the same names maintainers forgot to register. Slopsquatting subset.
Useful to note: An independent security firm (Aikido Security) had already identified some of the same vulnerable namespaces via full registry scanning and claimed depcruise as a defensive placeholder in July 2025 — discovered during this investigation via web search.
What did Anthropic say?
HackerOne #3682326 — filed as Critical (CVSS 9.0). Closed as Informative. Their framing, on record:

"A model proposing an unscoped package name where a scoped one exists is a model-output quality issue, and the underlying typosquatted npm package is third-party dependency hijacking — neither is a vulnerability in Claude Code's security controls."

The security team considered the permission prompt the relevant control — and by that framing, it functioned correctly. They also noted this is a topic of active interest to their model safety team, drawing a clear line between the security boundary (working as designed) and the model behaviour question (tracked separately). The model safety team was contacted separately via modelbugbounty@anthropic.com with the draft version of this writeup.
